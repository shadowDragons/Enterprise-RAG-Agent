# 企业内部知识库上传式 RAG 开发计划（AI 执行版）

## A. AI 必读硬规则

这部分优先级最高。AI 每次开始执行前，必须先读完本节。  
如果文档中部细节与本节理解冲突，以本节为准。

### A1. 最高依据

- `rag笔记.md` 是本项目一期实现的最高设计依据。
- AI 做方案选择、实现取舍、验收判断时，必须优先服从 `rag笔记.md`。
- 如果当前代码、README、旧实现与 `rag笔记.md` 冲突，在本期范围内优先向 `rag笔记.md` 靠拢。

### A2. 一期定位

- 本项目定位是：`企业内部知识库上传式 RAG 系统`
- 本期目标是：把当前项目从“课程 demo”升级为“可内部试点系统”
- 本期只做：`本地文档上传入库`
- 本期不做：`外部 connector`

### A3. 核心原则

- 不要把本项目当作“向量库 + prompt”的玩具系统。
- 必须把它当作“文档系统 + 检索系统 + 生成系统 + 评估监控系统 + 生产治理系统”。
- `chunk 质量比 chunk 数量更重要`，禁止靠盲目增大 top-k 或堆上下文掩盖检索问题。
- 检索主链路必须遵循：`hybrid retrieval -> 融合 -> rerank -> ACL 过滤 -> context assembly`
- 生成主链路必须遵循：`grounding -> citation -> abstention -> structured output -> 后验校验`
- 评估和观测必须与实现同步推进，不能等功能写完再补。
- 上线能力不等于功能跑通，必须考虑降级、恢复、失败模式和 readiness checklist。

### A4. 红线

- 如果一个方案会损害权限隔离，默认不合格。
- 如果一个方案会损害引用可信性，默认不合格。
- 如果一个方案会损害拒答能力，默认不合格。
- 不允许用“先能跑”作为长期保留明显错误方案的理由。
- 允许做过渡方案，但必须写明：
  - 这是过渡方案
  - 它违背了 `rag笔记.md` 的哪个点
  - 风险是什么
  - 以后怎么替换

### A5. 非本期目标

以下内容本期不要主动实现：

- 外部 connector
- webhook / polling / reconciliation
- 多租户
- Agentic RAG
- MCP / 多工具调用
- 大规模 A/B test
- LLM-as-judge 在线自动评估

## B. AI 执行流程

AI 每次开始一个里程碑时，必须按这个顺序做。

1. 先确认当前要执行的里程碑，只做一个里程碑范围内的事。
2. 先阅读 `rag笔记.md` 对应章节，再开始设计实现。
3. 先写出本次要遵守的 3-5 条核心规则，再动手。
4. 小步实现，不做一次性超大重构。
5. 每个能力完成时，同时补：
   - 测试
   - 文档
   - 最小可观测性
6. 提交前必须跑一次“提交前检查清单”。
7. 每次有实质进展后，必须更新本文件中的“里程碑状态表”和“进度日志”。

## C. 本期范围

### C1. 必须做

- 本地文件上传入库
- 检索链路生产化
- 权限预过滤
- 结构化回答与拒答
- 评估与观测

### C2. 明确不做

- 自动同步外部数据源

- 复杂实验平台

## D. 当前基线

### D1. 当前已具备

- 文档上传
- 异步入库雏形
- Qdrant dense 检索
- 多知识库选择
- LangGraph 澄清 / 审核工作流
- SSE 流式回答
- 审计日志、任务中心、系统总览

### D2. 当前主要缺口

- 没有真正的 BM25 / hybrid retrieval
- 没有检索阶段 ACL 预过滤
- 入库任务仍偏 demo 化

- 没有 golden dataset 和回归评测
- 没有限流、降级、上线检查表

## F. 里程碑详细要求

每个里程碑都按同样结构写。AI 执行时优先看 `阶段硬规则` 和 `禁止偷懒`。

### M0 范围冻结

#### 阶段硬规则

- 必须明确区分 `Demo RAG` 和 `生产级 RAG`
- 不要把“功能能跑”写成“可以上线”
- 必须写清本期不做 connector，不等于不知道 connector 的生产设计

#### 必须实现

- 更新 README 和相关文档
- 明确一期目标、二期目标、非本期目标
- 写清本项目是“企业内部知识库上传式 RAG”

#### 禁止偷懒

- 不要继续保留模糊表述，让人误以为近期会做 connector
- 不要把“以后可扩展”写成“当前已支持”

#### 验收

- 文档边界明确
- 一期与后续范围可直接读懂

### M1 检索主链路升级

#### 阶段硬规则

- 必须体现 `向量检索和 BM25 失败模式互补`
- 召回层目标是“别漏”，重排层目标是“挑得准”
- 最终进 prompt 的上下文不是原始 top-k，必须经过二次整理

#### 必须实现

- 增加 lexical retrieval
- 实现 dense + BM25 并行召回
- 实现融合，先做加权融合，保留 RRF 扩展点
- 保留并增强 rerank
- 记录 dense / BM25 / 融合 / rerank 各阶段 trace
- 做 context assembly：预算、排序、去重，必要时压缩

#### 禁止偷懒

- 不要继续停留在“dense 检索 + 简单词面打分”
- 不要把融合和重排混成一个黑盒步骤
- 不要把召回数量变大当作质量提升

#### 验收

- 专有名词、制度名、文件名类问题效果优于当前版本
- trace 能解释最终结果怎么来的
- 有检索回归测试

### M2 权限预过滤与知识治理

#### 阶段硬规则

- ACL 是硬约束，不是附加项
- 权限优先做预过滤，不要默认后过滤
- 元数据是检索、过滤、评估、治理的基础

#### 必须实现

- 一期先做知识库级 ACL
- 聊天工作流中传递用户可访问知识库范围
- KB 选择逻辑与用户授权范围求交集
- 检索前按授权范围过滤
- 补治理字段：`source`、`version`、`updated_at`、`tags`、`content_hash`

#### 禁止偷懒

- 不要只做接口级 RBAC 就宣称“权限已完成”
- 不要把文档版本、新鲜度问题推迟到以后而不留模型字段

#### 验收

- 未授权知识库不会进入检索候选集
- 同一请求在不同身份下结果不同且可解释
- 数据模型具备基础治理字段

### M3 入库工程化

#### 阶段硬规则

- ingestion 不只是“文件转文本”
- 去重是主链路要求，不是可选优化
- chunking 不能默认只有一套规则
- `chunk 太小会丢上下文，太大会稀释信号`
- chunk 必须带 metadata，不能是裸文本

#### 必须实现

- 将 `BackgroundTasks` 升级为更稳定的异步任务执行机制
- 支持幂等、失败分类、重试、状态查询
- 增加去重
- 增加 OCR
- 至少按 Markdown / 纯文本 / PDF 区分基础 chunking 策略
- 设计文档覆盖更新与重建策略

#### 禁止偷懒

- 不要把“只要能切出来”当作 chunking 完成
- 不要忽略 metadata、去重、版本替换策略
- 不要把扫描 PDF 继续当成“以后再说”，至少要接出扩展位或最小实现

#### 验收

- 批量上传时任务稳定
- 重复上传不会制造明显脏数据
- 扫描 PDF 可解析并进入检索

### M4 生成层可信度

#### 阶段硬规则

- prompt 是整个平台最关键的一段文本
- 必须落实 `grounding、citation、uncertainty、scope`
- 引用必须被强制
- 结构化输出不是锦上添花
- 证据不足时优先 `abstention` 或澄清

#### 必须实现

- 回答结构化，至少包含 `answer`、`citations`、`confidence`、`fallback_reason`
- 增加 citation 强制校验
- 增加低置信度拒答或澄清
- 增加基础安全过滤：敏感词、PII、secret redaction
- prompt 版本化，至少记录版本与变更说明

#### 禁止偷懒

- 不要只写一句“不要乱编”就算防幻觉
- 不要只让模型“尽量引用”，必须结构和流程上强制
- 不要因为本期不做全量 NLI，就完全不留后验校验扩展位

#### 验收

- 信息不足时优先拒答
- 回答结构稳定可消费
- 引用不匹配错误率下降

### M5 评估与可观测性

#### 阶段硬规则

- 评估不是报告，是发布流程的一部分
- 每次改 `chunking / retrieval / prompt / model` 都应触发回归
- retrieval 和 generation 必须分开观测
- dashboard 不能只看性能

#### 必须实现

- 建立 golden dataset
- 评估拆成 retrieval 和 generation 两层
- 支持关键改动自动回归
- 增加用户反馈入口
- 系统总览补充：延迟、空召回率、拒答率、审核率、命中率

#### 禁止偷懒

- 不要只做一个脚本输出分数却不接入开发流程
- 不要把用户反馈做成纯展示功能而没有后续数据沉淀

#### 验收

- 有固定评测命令与基线
- 每次关键改动能跑回归
- 能定位退化是在 retrieval 还是 generation

### M6 稳态运行与上线准备

#### 阶段硬规则

- 必须显式设计 `degradation ladder`
- failure handling 要覆盖典型失败模式
- readiness checklist 是硬门槛，不是形式文档
- 安全、权限、PII、secret redaction、审计日志都是红线能力

#### 必须实现

- 增加 rate limiting
- 增加超时、重试、故障降级
- 增加 degradation ladder，例如：
  - Level 0：完整链路
  - Level 1：关闭部分增强能力
  - Level 2：只用 dense 或只用 BM25
  - Level 3：只返回检索结果或明确降级提示
- 完善 readiness checklist
- 增加基础告警能力

#### 禁止偷懒

- 不要只考虑理想路径
- 不要把“服务挂了直接报错”当作故障处理
- 不要把 readiness 写成抽象口号

#### 验收

- 依赖异常时系统可退化
- 高负载或部分依赖失败时有明确 fallback 路径
- 上线前有标准化检查项

## G. 每次执行前自检

AI 开始任何开发任务前，必须先回答这 6 个问题：

- 当前任务属于哪个里程碑
- 对应 `rag笔记.md` 哪几课
- 本次最重要的 3 条硬规则是什么
- 本次最容易偷懒做错的地方是什么
- 如果做简化，实现风险是什么
- 本次交付如何证明自己没有偏离 `rag笔记.md`

- 我是否说明了这些简化的风险

## I. 进度记录硬要求

AI 每次执行任务后，必须记录进度。进度记录不是可选项。

### I1. 什么时候必须更新

- 完成一个里程碑中的一个子任务后
- 完成一次有实质内容的代码改动后
- 完成一次仅文档 / 方案 / 评估基线的推进后
- 发现阻塞并决定暂停当前任务时
- 切换到下一个里程碑前

### I2. 必须更新哪些内容

- `J. 里程碑状态表`
- `K. 进度日志`

### I3. 状态值规范

里程碑状态只能使用以下值：

- `not_started`
- `in_progress`
- `blocked`
- `completed`

### I4. 记录规则

- 不要覆盖旧日志，必须追加新记录。
- 如果当前只是做了分析、方案调整、文档推进，也要记进度。
- 如果状态是 `blocked`，必须写清：
  - 阻塞点
  - 当前影响
  - 建议下一步
- 如果状态从 `in_progress` 变为 `completed`，必须补一句“完成依据”。

## L. 一期完成标准

一期结束时，系统必须满足：

- 支持本地上传文档，稳定入库，支持 OCR
- 支持 hybrid retrieval，并有可解释 trace
- 支持知识库级 ACL 预过滤
- 支持结构化回答、citation 校验、低置信度拒答
- 具备 golden dataset 与基础回归评测
- 具备限流、降级、readiness checklist 的最小闭环
