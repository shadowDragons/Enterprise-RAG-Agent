# 一期范围说明

## 1. 定位

本项目一期定位是：`企业内部知识库上传式 RAG 系统`。

这里的关键词只有三个：

- `企业内部知识库`
- `上传式`
- `RAG 系统`

它不是外部数据连接平台，不是多租户知识中台，也不是 Agent 工具编排平台。

## 2. 最高依据

一期范围和方案选择，按以下优先级执行：

1. [rag笔记.md](../rag笔记.md)
2. [企业内部知识库上传式RAG开发计划_AI执行版.md](./企业内部知识库上传式RAG开发计划_AI执行版.md)
3. 仓库当前代码与 README

如果旧文档、旧实现、旧口径与上述内容冲突，一期内优先向 `rag笔记.md` 和开发计划靠拢。

## 3. 一期要解决什么

一期目标是把当前课程 demo 升级为可内部试点系统，重点不是“再多加几个功能”，而是把下面几条主链路做扎实：

- 本地文件上传入库
- 检索主链路升级
- 知识库级 ACL 预过滤
- 结构化回答、引用和拒答
- 评估与可观测性
- 基础稳态运行能力

## 4. 当前基线不是一期完成态

当前仓库已经具备：

- 文档上传
- 异步入库雏形
- Qdrant dense 检索
- lexical rerank
- 多知识库选择
- LangGraph 澄清 / 审核工作流
- SSE 流式回答
- 任务中心、审计日志、系统总览

但当前仓库还没有完成一期承诺的关键能力：

- 真正的 BM25 / hybrid retrieval
- 检索前 ACL 过滤
- OCR、去重、版本治理
- 结构化回答强校验与低置信度拒答
- golden dataset 与回归评测
- 限流、降级、readiness checklist 闭环

因此，当前仓库可以被称为“一期基线”，不能被称为“一期已完成”。

## 5. 一期明确不做

以下内容不属于当前一期范围：

- 外部 connector
- 外部数据源自动同步
- 多租户隔离
- Agentic RAG
- MCP / 多工具调用
- 大规模 A/B test
- 在线 LLM-as-judge 自动评估

这不代表这些方向不重要，只代表它们不应该挤占当前一期的主链路建设。

## 6. 二期方向

二期可以在一期完成后继续考虑：

- 外部 connector 与同步编排
- 更细粒度权限体系
- 多租户隔离
- 更复杂的 Agent 工作流
- 更完整的实验平台与在线评估

二期方向只能写成“后续规划”，不能写成“当前已支持”。

## 7. 当前里程碑顺序

一期按以下顺序推进：

1. `M0` 范围冻结
2. `M1` 检索主链路升级
3. `M2` 权限预过滤与知识治理
4. `M3` 入库工程化
5. `M4` 生成层可信度
6. `M5` 评估与可观测性
7. `M6` 稳态运行与上线准备
